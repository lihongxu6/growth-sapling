# 需求框架 · 用户引导功能（草稿 v0.1 / 待确认）

> 本文件为「用户引导功能」的**需求规划框架草稿**，供讨论确认。
> 确认后正式写入 `prd-成长小树苗.md`（单文件 living PRD）的新版本章节，按 EARS 原则撰写。
> 关联决策：§3.15（首次轻引导 + 隐私透明提示推迟到下一版，从界面设计整体做）、#57（相册/摄像头隐私声明 + 授权弹窗）、§3.8（反感单薄内容多开独立页）、§3.10（不动物上线内容）。

---

## 1. 背景

- v2.0 已提交审核（2.0.1），含开屏页 + 成长页用户信息区（本地版，不接云）。
- 现状缺口：用户首次进入后，对「怎么用、数据存在哪、为何要相册/摄像头授权」缺少统一说明。
- §3.15 已决策：首次轻引导 + 隐私透明提示统一放到下一版「小程序使用引导规划」，**从界面设计整体做**，不零散塞进各功能。
- 2026-07-20 用户确认：**下一版本 = 用户引导功能**（首次轻引导 + 隐私透明提示）。

---

## 2. 目标

- **G-目标1**：用户首次进入即理解产品价值与核心操作路径（打卡 / 补卡 / 看成长）。
- **G-目标2**：把「数据仅存本机、不上传」透明融入引导，消除隐私疑虑。
- **G-目标3**：相册/摄像头授权与引导说明衔接，授权弹窗前用户已知晓原因。
- **G-目标4**：不打断既有已验证界面（§3.10 不动物上线内容；§3.8 反感单薄内容多开独立页）。

---

## 3. 用户与场景

- **主用户**：7–8 岁儿童 + 其家长（共同使用；家长负责初始设置与监督，儿童每日打卡）。
- **场景 A（冷启动）**：首次安装 / 清空 Storage 后进入，不知道从哪开始。
- **场景 B（授权疑虑）**：设置头像时弹出相册/摄像头授权，不理解为何要。
- **场景 C（数据疑虑）**：想知道打卡记录去哪了、会不会上传、换手机怎么办。

---

## 4. 功能清单（候选）

| 编号 | 功能 | 说明 | 优先级 |
|---|---|---|---|
| **G1** | 首次进入引导流 | 开屏页之后，1–3 屏轻引导讲清「打卡 / 补卡 / 成长」核心路径与价值 | P0 |
| **G2** | 隐私透明说明 | 引导流内嵌「数据仅存本机、不上传」说明，与授权场景衔接 | P0 |
| **G3** | 权限授权衔接 | chooseImage 前 `wx.requirePrivacyAuthorize` 弹窗，文案与 G2 一致（#57 待补代码）| P0 |
| **G4** | 非首次跳过 + 重看入口 | 已引导过（本地标记）不再弹；设置/关于里可「重新看引导」| P1 |
| **G5** | 操作型微引导（可选）| 高亮「今天第一个任务」「补卡入口」等，首次完成即消失 | P2 |

## 6. 关键假设（待确认）

1. **引导受众以谁为主**：家长（文字可稍多）还是儿童直接读（需更童趣/图文）？→ 决定文案语气与图文比例。
2. **G1 形态**：开屏后接 1–3 张独立引导卡（轻量覆盖层），还是直接在现有页面上做「指向式」微引导？后者更不打断（§3.8 倾向）。
3. **隐私说明节点**：引导流一次性说清，还是拆到「首次设置头像时」才弹授权说明（更贴合授权场景）？

---

## 7. 验收标准（草案）

- 首次进入出现引导，且可在 N 步内看完 / 跳过；
- 引导中出现「数据仅本机」说明；
- 设置头像时授权弹窗文案与引导一致；
- 已引导用户再次进入不弹，可从设置重看；
- 不新增独立页面（复用现有界面 / 覆盖层）。
