## 二、MVP 经验总结

### 2.1 工程方法论

#### ✅ 做得好的

1. **H5 先行，小程序后跟**：先在 H5 上完成全部功能并冻结设计稿，再移植到小程序。设计稿成为"唯一真相源"（SSOT），移植时有精确对版参照物。

2. **CI 编译链先通**：在写任何功能代码前先打通 `miniprogram-ci preview`，确保每改一段代码就能验证——这避免了"写完一堆代码才发现编译不过"的灾难。

3. **踩坑即沉淀**：每修一个 bug 就写一条 MEMORY.md 规则，30 条踩坑覆盖了 WXML 语法限制、TabBar 渲染机制、弹窗定位方案、事件冒泡等关键知识。

4. **提案制引入创意流程**：生成图标/头像等主观资产时，出 3 套方案让用户选，选后再实现。避免 AI "拍脑袋画"的返工。

#### ❌ 需要改进的

1. **Phase 1 骨架代码太糙**：为了快速搭建，CSS 变量、字号、圆角、背景色未严格对齐设计稿，导致 Phase 2 修复时发现了 12 处不一致。教训：**骨架阶段就应严格对齐 SSOT**。

2. **"修好了"但实际没修好**：有两次声称修复完成但用户截图显示未生效（#22）。根因是只 grep 了代码"已改"就交差，没有对照实际渲染验证。教训：**必须对照实际渲染（ffmpeg 切帧或用户截图）确认修复效果**。

3. **核心数据流最容易被漏掉**：`recalcBadges()` 是 H5 的核心数据流函数，但小程序 Store 完全遗漏了它（#28），因为"眼睛只看到 UI，脑子没想到数据流"。教训：**移植时必须显式列出每个 Store 函数并标注来源**。

### 2.2 微信小程序特有坑点

| 坑点 | 根因 | 解法 |
|------|------|------|
| WXML 不支持函数调用 | `{{fmtDate(viewDate)}}` 传的是函数引用，不是调用结果 | JS 中预计算为字符串再 `setData` |
| TabBar 图标只渲染中心 27×27 | 81×81 画布中只有中心区域可见 | 图标内容区填 54×54，核心图形在中心 27×27 |
| 弹窗稳定居中 | `position:relative + top/left` 不重置定位基准 | `position:fixed + transform:translate(-50%,-50%)` |
| 弹窗内滚动区域 | `scroll-view` 在 flex 中撑大导致标题被挤出 | `view + overflow-y:auto`，销毁重建时滚动自动归零 |
| 事件冒泡 | `bindtap` 会从子元素冒泡到父元素 | 子按钮用 `catchtap` 阻止冒泡 |
| IP 白名单 | 沙箱 IP 不固定，每次都可能变 | 开发阶段关闭 IP 白名单 |

### 2.3 协作效率

| 好的实践 | 效果 |
|----------|------|
| 分阶段交付验证 | 打卡页 → 统计页 → 任务页，逐页修复，不积压 |
| 视频录屏 + 截图 | 用户用录屏反馈问题，比文字描述精确 10 倍 |
| ffmpeg 逐帧分析 | 从视频提取关键帧对照代码验证，避免"修好幻觉" |
| MEMORY.md 跨会话记忆 | 每次会话先读记忆，不重复踩坑 |

| 序号 | 优化项 | 说明 |

| U1 | 统计页视频分析修复 | 用户已录制统计页和任务页的操作视频，需逐帧分析并修复问题 |

## 六、给下次迭代的建议

1. **骨架代码就要严**：不要"先写个大概再修"，第一版就对齐设计稿 SSOT
2. **编译检查不要省**：每改一个文件就跑 `node scripts/build.js preview`
3. **UI 改动先出方案**：涉及视觉/交互的改动，先查规范 → 出 3 套提案 → 用户选 → 再实现
4. **数据流不要漏**：移植时显式列出所有 Store 函数并标注 H5 来源行号
5. **验证看渲染不看代码**：修完 bug 后对照实际渲染确认，不要只 grep 代码
