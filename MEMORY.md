## 3. 用户偏好与决策（最重要）

### 3.1 流程纪律
- **严格按产研流程走**：原型 → UI 设计稿 → 技术方案.md → 代码 → 测试验证，每环评审通过才进下一环。
- **不许跳过任何一环**。用户明确反感"直接在原型上改样式"的做法——那导致质量差。
- **Harness Engineering**：每个交付物是下一阶段的精确输入锚点。
- **Route A**：无 Figma，沙箱内出高保真 HTML/CSS 视觉稿 + 视觉规范文档。不引入外部账号。
- **整体路线一次规划、分步执行**：本轮只做当前环，下一环需用户确认再启动。

### 3.2 沟通偏好
- 每次询问用户的问题 ≤ 3 个。
- **业务决策必须先给方案 → 用户确认 → 再执行，绝不可跳过确认直接改代码。**
- **代码修改两步走**：① 本地修改（不改 index.html），用户确认效果；② 用户点头后才 commit + push 上线。
- 改代码/文档后必须提供 `git diff` 给用户核对。
- 每次提交附 commit message + push，线上 Demo 同步更新。
- **每个阶段完成后，主动提醒用户将成果流转给下一环节。**
- **环境调试适可而止**：技能行为已通过 MEMORY.md 在本项目生效，不纠结沙箱/客户端加载细节；遇到环境类问题先判断是否阻塞主线，不阻塞则跳过（详见会话记录：新会话 git 连接失败，但当前会话正常、文件已 push 到 GitHub，无需纠缠）。
- **创意产物也需两步确认**（2026-07-12 新增）：生成图像、文案、命名、配色等"看起来是改进"的产物，属于业务决策。流程：① 生成多版本给你看 → ② 你选/给反馈 → ③ 仅在你点头后才 commit + push。**绝不"觉得是改进就直接提交"**。

- **火焰图标**：今日无完成 = 🪵；今日有完成 = 🔥 发光。

### 3.4 技术决策（2026-07-11 技术方案评审确认）
- **连续天数截止**：按自然日 00:00 重置，非 24h 滚动窗口
- **emoji 切手绘**：不紧急，待插画师交付后替换文件即可
- **技术细节**（ID 生成策略 / 低端机阈值 / 分包策略）：AI 在实现时自行决策，无需用户纠结

- **自动同步**：使用微信官方 miniprogram-ci（CLI 工具），沙箱内编译 + 生成预览二维码，无需用户安装 DevTools。

### 3.5 研发阶段质量约束（2026-07-12 新增）

> 以下规则适用于小程序原生代码开发阶段。每次编写/修改代码文件时，AI 必须逐条自查。

- **QC1 多文件一致性**：修改数据模型（如 `mgmtTasks` 结构）时，必须同步更新所有引用该数据的页面 JS 文件。
- **QC2 WXML/WXSS/JS 三件套**：每个页面必须包含同名三件套，且 `app.json` 中 `pages` 数组已注册。漏注册 = 页面不存在。
- **QC3 编译自查**：每完成一个 Phase，必须运行 `miniprogram-ci` 编译检查，编译未通过前不允许进入下一 Phase。
- **QC4 数据绑定完整性**：WXML 中 `{{ }}` 引用的变量必须在对应 JS 的 `data` 中声明。setData 的 key 必须与 WXML 绑定名一致。
- **QC5 交互逻辑回归**：移植 H5 代码时，必须对照 MEMORY §5 交互逻辑清单逐条验证，确保无回归。
- **QC6 交付前检查清单**：每次交付给用户验证前，AI 必须跑完编译检查 + 交互逻辑清单回归，并在回复中报告结果。
- **QC7 微信官方文档优先**：涉及小程序 API/组件/限制的判断，必须基于官方文档，不凭训练记忆。

### 3.6 文案决策（2026-07-16 确认）
- **未来日期 banner 文案**：选未来日期时，打卡页顶部 banner 显示「⏰ 不要着急，到这天了才可以打卡哦」（⏰ 图标）；历史补卡保持「📅 正在为 X月X日 补卡」。点击未来日期任务卡片的 toast 文案同步一致。
- 根因：`isBackfill` 原逻辑 `!isToday(viewDate)` 把未来日期也判为补卡模式，导致 banner 误显示。
- 修改文件：`miniprogram/pages/home/home.js`（isBackfill 逻辑 + toast 文案）、`miniprogram/pages/home/home.wxml`（banner 加 `wx:if="{{isFuture}}"` + `wx:elif="{{isBackfill}}"` 分支）。

- 与既有规则一致：本文件顶部「读取/写入规则（强制）」为最高优先级；§3.2 业务两步确认、§3.10 不动物上线内容等纪律须长期坚持。

### 3.14 避免为低频需求过早付费（产品收益视角，2026-07-19）

- 用户明确的产品判断：**微信云开发的核心价值 = 同步微信名 + 跨设备不丢数据**；在没有真实用户、且真实用户跨设备需求很低频时，**完全没必要承担每月云成本**（19.9 元/月）。
- 决策：**v2.0 账户体系（云开发跨设备同步）明确推迟**，等「真实用户出现跨设备同步需求」时再开通云环境（届时付费合理，或借未上线态的 6 个月免费窗口）。
- 当前 v2.0 聚焦：开屏页（已完成）+ 成长页用户信息区（本地版，不接云）。
- 原则：避免 premature optimization / 为低频或未发生的需求预付成本；先验证需求真实性，再投入资源与合规成本。

### 3.15 成长页本地版用户信息区·决策（2026-07-19 本会话确认）

- **首次引导**：本地版**不做**首次轻引导；仅占位（🌱 + "点击设置昵称"），用户想设再设。原因：首次引导留待后续版本**统一做整个小程序的引导功能**（用户明确决策）。
- **老设备降级（概念澄清）**："降级"指基础库 < 2.21.2 时 `chooseAvatar`/`input type=nickname` 原生组件不可用，需 fallback（plain text input + 默认头像）。**≠ 本地存储**——wx.setStorageSync 是既定方案，不是降级对象。推荐方案 A：不额外做降级，要求基础库 ≥ 2.21.2（近全覆盖、最简单、契合 #3.14 避免 premature complexity）；**用户已确认方案 A（2026-07-19）**。
- **隐私声明（概念澄清 = 两件事）**：① 微信公众平台《隐私保护指引》后台声明（告诉平台收集啥，仅当调用需声明接口如 getUserProfile/位置等才强制）；② 应用内透明提示文案（自己写，不依赖 API）。**决策：采纳用户提议 = 应用内加一句"打卡数据仅保存在你手机本地，不会上传"透明提示（儿童产品加分）；平台隐私指引本期不额外声明（与 v1.0 一致，#41 已确认"未采集隐私"，本地版未调用 getUserProfile 等需声明接口）**。等将来接云账户再补声明。
- **微信安全检测时机（澄清 Q2/Q3）**：头像/昵称的安全检测发生在**用户选择/输入的当下**，由微信原生组件透明调用其服务端（头像 mediaCheckAsync / 昵称 msgSecCheck，2.24.4+）；**不是我们存本地时触发**——`setStorageSync` 时内容早已过检。我们**不写任何后端、不调接口**，与“不用云开发”完全不冲突。`wx.setStorageSync` 是纯前端本地存储 API（本机沙箱、零网络请求），用户“理解完全正确”。

- 与 tech-spec §2.0.6（云版）关系：本地版是其推迟期派生实现；用户确认 3 项后写入 tech-spec §2.0.7。

### 3.16 角色切换规则（2026-07-19 用户明确）

- 本项目“一人+AI”协作中，Sophia 角色随阶段切换：**阶段三（设计与研发评审）= 架构师**，负责技术方案 / 研发设计方案；**设计方案过审后 = 代码开发工程师**，负责落地执行。
- **技术任务拆分属于「研发设计方案」**，是架构师（当前阶段）的产出，随后由工程师执行——即任务拆分写在 tech-spec 的实施方案章节（如 §2.0.7），不另起炉灶。
- 含义：当前以架构师身份产出含任务拆分的研发设计方案 → 用户确认定稿 → 切工程师角色按拆分实现 → 编译 / 真机验证 / 回归。

### 3.17 身份衔接 · 交付物与标准框架（2026-07-19 用户明确）

- **用户自述角色（铁律）**：只负责「需求提出 + 过程监督 + 决策」。代码/文档的具体产出与衔接由 Sophia 全权安排，用户不介入实现细节。
- **无缝衔接机制（为何能无缝）**：架构师与工程师两身份共享同一份上下文——MEMORY.md（全程记忆）+ tech-spec §2.0.7（研发设计方案 = 交接契约）。架构师交付 §2.0.7 经用户确认 commit 后，工程师身份直接读它开工，无需重复沟通。纪律兜底：每步 MEMORY 更新+commit/push（可追溯）+ 两步确认（方案→确认→改→确认→上线，§3.2）。
- **交付物与标准**：
  - 架构师（阶段三）：交付 = 研发设计方案（EARS 需求 + 任务拆分 T1–T9 + 微信官方约束核验 #17/#QC7 + 验收标准）。标准 = 用户确认定稿 + 写入 tech-spec 并 commit+push 到资料库。
  - 工程师（阶段四+）：交付 = ① 代码变更（stats.wxml/js/wxss + utils）② compile 报告 ③ 真机预览二维码 ④ 回归报告。标准 = miniprogram-ci compile ✅；真机扫码验证 ✅（原生组件必须真机，DevTools 不作判定 #22/#34）；对照 design-spec §8.1 / MEMORY §5 L-list 回归 = 下方统计块零变化（QC5/QC6）；用户扫码体验确认。
  - 用户监督清单（只盯 4 条）：① 可追溯（每步 MEMORY 有更新 + GitHub 有 commit）；② 两步确认（改动先给方案→确认→改→再确认→上线）；③ 三项达标（compile/真机/回归）；④ 决策权在用户（是否 push 体验版 / 是否提审 / 范围是否调，用户拍板，AI 不擅自动）。
- **透明提示文案候选（待用户选，用户-info 块内昵称下方一行小字）**：A「📱 数据只存在这台手机，不会上传」（推荐）/ B「打卡记录仅存本机 · 不会上传网络」/ C「你的成长数据都在本机，放心用～」。
